    Refactoring the middleware's Mediarender to control multiple Player instances

    Mediarenderer is the middleware's entry point for playing media (UI or other clients do NOT directly send messages to the player/recorders component). It is also a kind of abstraction (implementing an API inspired by and very close to the UPnP standards, regarding Audio/Video). The two main goals of this decoupled architecture are:
    • Provide a comprenhensive but simple API to the client (typically UI), hiding the details in the depth, and,
    • Allow the player component implementation to focus on the rendering of the media itself, whereever it comes from, allowing a simple and design of this critical component, which helps to ensure a good quality and reliability.

    Context

    Historically, to simplify the Mediarender's implementation, the assuption was made that only ONE media would be played at a given time (whatever its source, live channel or a video on USB drive), and then, only ONE player instance was needed. But, with the increasing power of processors, controlling multiple player's instances became realistic, enabling features like:
    • Mozaic (wanted by Sky Italia): by displaying multiple channels on screen at the same time (before, operators had to generate mozaic stream on the backend and stream it on a dedicated channel),
    • Fast Zapping (wanted by Canal+): a number of channels (typically: previous and next but it's not limitative and a client/U.I. choice) are preloaded, meaning all the rendering chain is running for each of those channel, but they're not displayed on screen. When zapping (typically using "Progam +/-" keys) on a preloaded channel, mediarenderer just switch the displayed channel, drastically optimizing the latency.

    Workload

    This implied modifications at multiple levels:
    • Interprocess Communication API (on DBus)
    • on the player itself
    • on the recorders mecanism (recorders are always instanciated when playing a live media, to allow timeshift, even if the user does NOT want to stores a "record" of the media)
    • and finally, on mediarender, to issue those DBus API calls on players and recorders, and manage their lifetime cycles.

    Achievements

    Mananing the player instances was pretty straigthforward ; the main complexity was about the recorders, how they bind (and detach) to player instances. Then my task were:
    • Analyse the full mediarender code (I had only a partial knowledge of it before)
    • Propose a refactored architecture for the component to the team (and to the maintener of mediarender), detailled as UML class and sequence diagrams
    • Implement and
    • test the code (manually and by fixing or adding unit tests)

Alten's Software Developmet Center

The S.D.C. was a division of Alten Rennes, which was aimed to provide a comptetive external consulting service (Technical Assistance), compared to Alcatel's indian subsidiaries. The arguments were:

Context

Alcatel's (nowadays, Nokia's) Open Service Platform (OSP) is a multi-blade server technology, powered by Solaris. Its role is to manage the classical commutated telephone network, to implement what is called the Intelligent Network. It allow valuable services far way beyond reaching the receipient of a voice call (which is the basic task of PTSN and does not imply I.N.).

I.N. products examples

Enterprise Communication at Network (ECN) is one of the main Alcatel's Intelligent Network product, dedicated at first to companies that spread around multiple locations but wanted to take advantage of the benefits of an on-site PABX, like short numbers to reach another corporate, but spreading across all site of the corporation.

By the time, complexity grew up, and it now allows fixe/mobile convergence (corporate people can seamlessly initiate a short-number call from their mobile phone, off-site...

It also allows the corporate user to manage its own mobility, setup call routing (e.g. to office, mobile, home or voicemail) from a public web interface (obv. using authentication).

Another I.N. product, Flexible Routing and Charging (FRC) allows, for example, a victim of a break down on the motorway, to call a special number (nationally unique for a given Automobile Brand) from its mobile and reach the nearest Technical Center, that is open, and having room for this customer.
